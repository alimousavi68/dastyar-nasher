گزارش جامع ممیزی کد و تحلیل معماری افزونه دستیار ناشر (Dastyar Nasher)
مقدمه
پیرو درخواست بررسی دقیق و عمیق ساختار کد، معماری، امنیت و کارایی افزونه «دستیار ناشر»، یک ممیزی کدنویسی (Code Review) کامل صورت گرفت. در این گزارش، تمام نقاط ضعف معماری، باگ‌های عملیاتی و مشکلات امنیتی به صورت ساختاریافته استخراج گردیده و همراه با راهکارهای فنی، بر اساس اولویت ریسک (از بحرانی تا پایین) دسته‌بندی شده‌اند.

مشکلات و نقاط ضعف شناسایی شده (بر اساس اولویت ریسک)
🔴 ریسک بسیار بالا (بحرانی - امنیت و معماری پایه)
۱. آسیب‌پذیری بحرانی دور زدن احراز هویت و دسترسی مستقیم (Broken Access Control & Unauthenticated Access)
محل وقوع: فایل‌های inc/scraper/index.php و inc/functions_loader.php
نوع مشکل: امنیتی / ضعف معماری پایه
تأثیر: این فایل‌ها با فراخوانی مستقیم و اجباری هسته وردپرس (require_once('../../../../wp-load.php')) به گونه‌ای طراحی شده‌اند که به صورت مستقیم از طریق مرورگر (Endpoint خارجی) قابل اجرا باشند. در فایل اسکرپر، با بررسی ساده $_POST['action'] == 'publish_scraper' بدون هیچ‌گونه چک کردن هویت کاربر (current_user_can) یا تایید توکن امنیتی (Nonce)، پست جدید در سایت ایجاد می‌شود. یک مهاجم می‌تواند به سادگی با ارسال درخواست‌های متوالی POST، پایگاه داده شما را پر از پست‌های جعلی کرده و منابع سرور را به شدت درگیر کند.
راهکار بهبود عملی:
گام اول: تمام کدهای require_once مربوط به wp-load.php و wp-admin/includes باید از فایل‌ها کاملا حذف شوند.
گام دوم: پردازش درخواست باید با استفاده از اکشن‌های استاندارد wp_ajax_* وردپرس انجام گیرد.
کد اصلاحی:
php

add_action('wp_ajax_publish_scraper', 'handle_publish_scraper_ajax');
function handle_publish_scraper_ajax() {
    // ۱. بررسی Nonce امنیتی
    check_ajax_referer('scraper_nonce_action', 'security');
    // ۲. بررسی سطح دسترسی مدیر
    if (!current_user_can('edit_posts')) {
        wp_send_json_error(array('status' => false, 'message' => 'عدم دسترسی مجاز'));
    }
    // ۳. دریافت مقادیر و اجرای اسکرپر...
}
۲. آسیب‌پذیری CSRF و پاک‌سازی تصادفی پایگاه داده (Cross-Site Request Forgery)
محل وقوع:
inc/functions/post_schedule.php (خطوط ۴ تا ۱۳: پاک‌سازی دستی آیتم‌های صف انتشار delete_item)
admin-pages/feeds_list.php (خطوط ۷۵ تا ۷۷: عملیات پاک کردن کل فیدها action=delete_all)
inc/functions/metabox.php (خط ۴۱: ذخیره وضعیت زمان‌بندی در متا باکس بدون اعتبارسنجی)
نوع مشکل: امنیتی
تأثیر: توابع حیاتی مانند پاک‌سازی جدول فیدها (remove_all_feed_on_feeds_table) بدون تایید توکن امنیتی (Nonce) فراخوانی شده‌اند. از آنجایی که شرط روی $_GET['action'] == 'delete_all' برقرار است، اگر مدیر سایت به صورت ناخواسته روی لینکی از منبعی خارجی کلیک کند (حتی در یک تصویر پنهان)، تمام فیدهای دیتابیس پاک خواهند شد.
راهکار بهبود عملی:
در صفحات ادمین هنگام ساخت لینک‌های حذف، از تابع wp_nonce_url استفاده کنید.
پیش از اجرای منطقِ حذف در فایل پردازشگر، حتما دستور if (!isset($_GET['_wpnonce']) || !wp_verify_nonce($_GET['_wpnonce'], 'delete_action_name')) { wp_die('درخواست نامعتبر'); } را اضافه کنید (همانطور که در db.php برای برخی بخش‌ها به درستی رعایت شده بود).
🟠 ریسک بالا (عملکرد، پایداری و جریان منطقی)
۳. ارسال درخواست HTTP کُند کننده در هوک ورود کاربران (Performance Blocker)
محل وقوع: inc/functions/license.php خط ۱۴۹ add_action('wp_login', 'wp_scheduled_delete_comments', 10, 2);
نوع مشکل: افت شدید عملکرد (Performance) و تجربه کاربری (UX)
تأثیر: با هر بار لاگین مدیر یا کاربران به سایت، افزونه متوقف می‌شود تا یک درخواست HTTP همگام (Synchronous) با سرور مرجع (sardabir.online) برقرار کند و اعتبار لایسنس را بسنجد. این فرآیند علاوه بر ایجاد تاخیر چند ثانیه‌ای آزاردهنده در هنگام ورود، باعث می‌شود اگر سرور سردبیر کند یا قطع باشد، عملاً فرآیند ورود به پنل وردپرس نیز مختل گردد.
راهکار بهبود عملی:
این بررسی نباید به wp_login قلاب (Hook) شود. استفاده از اکشن روزانه کرون (daily) که در محیط ناهمگام (Background Task) اجرا می‌شود بهترین راه است. در صورت نیاز ضروری به فراخوانی حین ورود کاربر، باید پارامتر 'blocking' => false به توابع wp_remote_post در بدنه تابع لاگین اضافه شود تا پردازش متوقف نگردد.
۴. استفاده اشتباه از بافر خروجی در حین ثبت پست (Output Buffering Misuse)
محل وقوع: inc/scraper/index.php (خطوط ۱۹۷ تا ۲۰۱)
نوع مشکل: خطای کدنویسی و پایداری
تأثیر: در این خطوط کدهایی مانند ob_start(); و ob_flush(); به اشتباه حول تابع wp_insert_post احاطه شده‌اند. در صورت وجود خطا در پردازش، این موضوع می‌تواند به خطای مرگبار Headers already sent منجر شود و مانع از پردازش صحیح یا هدایت (Redirect) در دیگر بخش‌های وردپرس شود.
راهکار بهبود عملی:
توابع ob_start و ob_flush کاملاً بی‌ارتباط با عملکرد درج پست در وردپرس هستند و باید به طور کامل از این بخش حذف شوند. بررسی خطا باید از طریق خروجی تابع صورت پذیرد (if (is_wp_error($post_id))).
۵. نشت حافظه (Memory Leak) در پردازش‌گر HTML
محل وقوع: inc/scraper/index.php و استفاده از simple_html_dom
نوع مشکل: عملکردی و مصرف بالای منابع سرور
تأثیر: کلاس simple_html_dom به داشتن نشت شدید حافظه معروف است. هنگامی که یک آبجکت HTML واکشی می‌شود، در صورت عدم پاک‌سازی اصولی، تا پایان اجرای پردازش در حافظه رم (RAM) سرور باقی می‌ماند. اگر افزونه در حال واکشی چندین فید به طور همزمان باشد، سایت با ارور Fatal Error: Out of memory مواجه خواهد شد.
راهکار بهبود عملی:
همواره در انتهای تابع scrape_and_publish_post متد آزادسازی حافظه را فراخوانی کنید:
php

if (isset($html) && is_object($html)) {
    $html->clear();
    unset($html);
}
🟡 ریسک متوسط (قابلیت نگهداری و استانداردهای توسعه)
۶. لود مستقیم و غیراصولی فایل‌های استاتیک در پنل وردپرس (Conflict Inducer)
محل وقوع: admin-pages/feeds_list.php و سایر صفحات مدیریت (مانند خطوط ۳۶ تا ۴۱)
نوع مشکل: تخطی از استانداردهای وردپرس و خطر تداخل
تأثیر: قرار دادن تگ‌های <script src="..."> و <link rel="stylesheet"> مستقیماً درون تگ‌های HTML صفحات ادمین یک اقدام بسیار غیر استاندارد (Anti-Pattern) است. این امر باعث لود مجدد نسخه‌های متضاد jQuery، Select2 و Bootstrap شده و می‌تواند ظاهر سایر افزونه‌ها یا امکانات خود هسته وردپرس را مخدوش کند.
راهکار بهبود عملی:
تمامی استایل‌ها و اسکریپت‌ها باید الزاما با استفاده از هوک admin_enqueue_scripts لود شوند:
php

add_action('admin_enqueue_scripts', 'cop_enqueue_admin_assets');
function cop_enqueue_admin_assets($hook) {
    if ($hook != 'toplevel_page_publisher_copoilot') return;
    wp_enqueue_style('cop-bootstrap', COP_PLUGIN_URL . '/assets/css/bootstrap.rtl.min.css');
    wp_enqueue_script('cop-sweetalert', COP_PLUGIN_URL . '/assets/js/sweetalert2@11.js', array(), false, true);
}
۷. باگ محاسباتی در منطق تقسیم و تعیین فواصل زمانی
محل وقوع: inc/functions/post_schedule.php (تابع calculate_post_publish_time)
نوع مشکل: باگ محاسبات ریاضی (احتمال خطای Division By Zero)
تأثیر: در خطوط انتهایی این تابع، محاسبات پیچیده‌ای مانند $post_per_hours = ($post_publisher_daily_post_count / ($interval / 3600)); انجام می‌شود. اگر $interval برابر ۱ باشد، مخرج کسر فوق العاده کوچک شده و نتیجه به شدت بزرگ می‌گردد. این موضوع باعث اختلال جدی در محاسبه دقیق فواصل بین انتشار اخبار می‌گردد.
راهکار بهبود عملی:
منطق محاسبه باید ساده‌سازی شود. فاصله هر پست بر حسب ثانیه برابر است با: (اختلاف ثانیه‌ای ساعت شروع و پایان روزانه) تقسیم بر (تعداد هدف اخبار روزانه). نیازی به درگیر کردن محاسبات ساعتی نیست.
php

$interval_seconds = max(1, $end_seconds - $start_seconds);
$post_publishe_interval_time = round($interval_seconds / max(1, $post_publisher_daily_post_count));
return $post_publishe_interval_time;
🟢 ریسک پایین (تجربه کاربری، سئو و کدهای تمیز)
۸. کوئری‌های پایگاه داده بدون Index گذاری و واکشی سراسری
محل وقوع: admin-pages/feeds_list.php و inc/db.php
نوع مشکل: بهینه‌سازی دیتابیس (Optimization)
تأثیر: در جدول custom_rss_items از دستور SELECT * برای شمارش استفاده شده (البته در برخی خطوط SELECT COUNT(*) نیز قرار دارد). مهم‌تر از آن، ستون‌های پر جستجو مانند resource_id و guid هیچگونه کلاستر ایندکسی (Index) ندارند. با بالا رفتن تعداد فیدها، سرعت لود صفحه لیست فیدها افت خواهد کرد.
راهکار بهبود عملی:
در فایل db.php در دستور CREATE TABLE، عباراتی مانند KEY resource_id (resource_id) و UNIQUE KEY guid (guid(191)) اضافه شود تا خواندن دیتابیس با سرعت بالایی انجام شود.
۹. عدم پشتیبانی از ساختار چند زبانه (Hardcoded Strings)
محل وقوع: تمام فایل‌های پروژه
نوع مشکل: استاندارد بین‌المللی‌سازی (i18n)
تأثیر: کلمات و رشته‌های فارسی مستقیماً درون کد PHP یا HTML نوشته شده‌اند که اجازه ترجمه یا تغییر گروهی ادبیات افزونه را در آینده نمی‌دهد و مغایر با استانداردهای ثبت افزونه در مخزن وردپرس است.
راهکار بهبود عملی:
از توابع ترجمه استاندارد وردپرس استفاده کنید؛ به عنوان مثال به جای 'تنظیمات دستیار' از __('تنظیمات دستیار', 'dastyar-nasher') استفاده شود.
🚀 پیشنهادهای جامع برای ارتقای روال توسعه و پیشگیری از تکرار
مهاجرت به معماری شی‌گرا (OOP) و فضاهای نام (Namespacing): بیشتر توابع پروژه در Global Namespace قرار گرفته‌اند (نظیر complete_url، encodeUrl و ...). این توابع عمومی به شدت مستعد تداخل و ایجاد Fatal Error در کنار افزونه‌ها یا قالب‌های استاندارد دیگر هستند. کدها باید در قالب کلاس‌های استاتیک یا فضاهای نام (Namespace) اختصاصی (مثلاً namespace Dastyar\Scraper;) کپسوله شوند.

حذف معماری Endpoint مبتنی بر فایل و لود مستقیم هسته وردپرس: قرار دادن یک اسکریپت PHP با دسترسی عمومی درون یک پوشه (مانند inc/scraper/index.php) که به تنهایی wp-load.php را با ساختار روت محلی بارگذاری کند، شیوه‌ای کاملا منسوخ و به شدت ناامن است. سیستم فعلی باید کاملا بر روی WordPress REST API بازنویسی شود.

ارتقای سیستم لاگ و عیب‌یابی (Logging System): استفاده مداوم از error_log در فایل‌های crons.php و post_schedule.php باعث پر شدن فایل لاگ سرور در Production می‌شود. پیشنهاد می‌شود یک کلاس Logging حرفه‌ای پیاده‌سازی شود که امکان فعال/غیرفعال‌سازی ذخیره گزارشات را از طریق تنظیمات صفحه ادمین به کاربر بدهد.

تأییدیه و بهداشت ورودی و خروجی‌ها (Sanitization & Escaping): تقریبا تمام مقادیری که از پایگاه داده واکشی شده و یا توسط کاربر ثبت می‌شوند ($_POST و $_GET) به صورت خام ذخیره و چاپ می‌شوند. استفاده اجباری از توابعی نظیر sanitize_text_field در ورودی‌ها و esc_html در زمان چاپ خروجی‌ها در فایل‌های پنل مدیریت الزامی است تا از باگ‌های خطرناک XSS پیشگیری شود. پیشنهاد می‌شود استاندارد WordPressCS را به ابزار PHP_CodeSniffer محیط توسعه اضافه کنید.
